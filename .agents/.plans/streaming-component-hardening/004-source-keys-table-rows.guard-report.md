## Spirit

The plan's `Why this matters`: in source-backed/streaming rendering, table body cells fell through to object-identity fallback keys, so a re-parse in the streaming tail re-mounted the entire `tbody` each flush instead of only the changed cell. The diff delivers this: `assignSourceKeysToChildren` now walks `node.rows` and assigns each body cell a stable `src:` key at plain `absoluteOffset` — row-index-independent, so an unchanged cell survives a sibling cell's growth (the new `toBe` DOM-identity test proves it, tagging the cell with a probe attribute that survives the reparse). Execution also surfaced a constraint the plan missed: `getStableRowKey` (`render-metadata.ts:358-362`) derives a keyless row's key from its first cell, so keying every row's first cell identically would have collapsed all rows to duplicate keys — the executor correctly gave each row its own explicit key, and the append-row identity test confirms rows stay mounted. Intent is met, not merely the checklist.

## Residual risk / follow-ups

- **Position-based row key.** The row key encodes `rowIndex` (`src:${absoluteOffset}:row:${rowIndex}`), so a row _inserted before_ an existing row in the source-backed path remounts it (and its cells). This is inherent — table rows have no meaningful per-row source offset (the plan's own offset caveat) — and streaming normally **appends** rows (the covered case) rather than inserting. Not a regression; if it ever matters, a source-backed row-inserted-before identity test would pin it. Recorded in the plan's maintenance notes.
- **Header/body-row-0 base overlap** (`src:${absoluteOffset}` shared) is harmless: header cells and body cells render in separate `{#each}` blocks, so keys only need within-block uniqueness, which holds.
- **Checkpoint 1 correction.** My first checkpoint wrongly recommended reverting to the plan's plain-`absoluteOffset` shape; I had missed `getStableRowKey`'s first-cell derivation. Corrected at Checkpoint 2 and reflected in the amendment. The log is append-only, so the error and its correction both stand as record.
