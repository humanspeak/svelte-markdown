## Spirit

The plan's `Why this matters` is that the **prop-driven** streaming path — the documented common case where `source` grows one LLM token at a time — ran `syncStreamingSourceFromProp → applyStreamingSource` synchronously on every prop change, so five same-frame updates meant five parse+diff+render passes, four of them invisible. The diff delivers exactly that intent and nothing adjacent to it: the append-only branch of `syncStreamingSourceFromProp` now parks the latest full source in `pendingStreamFullSource` and schedules a single rAF flush on the _existing_ `streamFlushHandle`, collapsing N same-frame appends to one `lexAndClean` (proven by T1 asserting 0-synchronous-then-1-after-flush, not merely "≤ N"). It resists the two ways this optimization usually goes wrong: last-write-wins coalescing keeps the _latest_ full source rather than accumulating stale deltas, and every non-append transition (reset/shrink/token-array) cancels the pending prop flush so a stale flush can't overwrite a reset — the reset-race the plan flagged as "where batching bugs hide," covered by T2/T3. No gamed criteria: the test file is byte-identical to the red-phase commit (`git diff 57c82b2 HEAD -- SvelteMarkdown.test.ts` empty), so the four assertions flipped green purely because the source changed.
