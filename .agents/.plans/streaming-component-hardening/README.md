## New findings not previously tracked as issues

These came out of this audit and were **not** open issues before:

- **001** — default-config XSS via the inline-HTML fast path (`Parser.svelte:221-226`).
  Confirmed end-to-end during planning (marked emits `<script>` as an html token;
  `script`/`style`/`object`/`meta`/`base` are absent from the `Html` map, so the
  `=== Html[tag]` identity check is `undefined === undefined`). **Highest priority.**
- **002** — offset-mode `' '.repeat(gap)` unbounded allocation (`SvelteMarkdown.svelte:306-316`).
- **003** — per-flush list/cell re-spread in `shrinkHtmlTokens` (`token-cleanup.ts:558-581`).
- **004** — missing `rows` walk in `assignSourceKeysToChildren` (`render-metadata.ts:247-251`).
- **008** — ~190 lines of dead legacy HTML code + a shipped bench module.
- **009/010** — lint/coverage/engines/test-coverage gaps.

## Findings considered and rejected (do not re-audit)

- **Token-cache 32-bit FNV-1a key collision serving wrong tokens** (`token-cache.ts`):
  real but statistically negligible at `maxSize: 50` + 5-min TTL, and a documented
  speed tradeoff. Not worth a plan unless cache size grows substantially.
- **`STREAM_BATCH_MAX_CHARS` mid-frame flush defeating batching** (from #327): a
  genuine tension, but the char cap bounds worst-case latency; raising/removing it
  is optional and folded as a note in plan 005, not a standalone plan.
- **Parser component-per-token (2× instances per block token)** (`Parser.svelte`):
  real architectural cost, but effort L / risk MED with unclear payoff given the
  existing leaf-inlining + keyed-each reuse. Deferred — revisit only if profiling
  shows block-token allocation dominating.
- **Streaming batch tests assert internal `lexAndClean` call counts** (white-box):
  brittle but partly intentional as an append-only perf contract; not worth a
  dedicated plan (plans 005/007 touch these tests where needed).
- **PARSER-01 Vite 8 rolldown self-import (external PR #323)**: `Parser.svelte:82`
  does self-import, and Vite 8 is present. Not planned here because plan **011**
  removes the reuse module and may restructure recursion anyway, and the breakage
  is claimed by an external PR, not reproduced in this read-only audit. If PR #323
  is being evaluated separately, that is the venue; revisit if 011 doesn't touch
  the recursion mechanism.
